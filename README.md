# Raspberry Pi 4 Buildroot Repository

This repository is set up to build a bootable Raspberry Pi 4 64-bit image using Buildroot.

## File descriptions

### Repository files

- `README.md`: Project overview, setup, and build instructions.
- `Makefile`: Main entrypoint for cloning Buildroot, applying config, and building images.
- `configs/pi4_64_defconfig`: Buildroot defconfig for Raspberry Pi 4 (64-bit).
- `br2_external/`: Custom Buildroot external tree containing local packages (including `av_services`).
- `.gitignore`: Ignores local Buildroot checkout and build outputs.
- `.gitmodules`: Tracks git submodule mappings.

### Generated directories

- `buildroot/`: Upstream Buildroot source tree cloned by `make buildroot`.
- `output/`: Build output directory (toolchain, target rootfs, logs, and images).

### Integrated submodules

- `external/av_services/`: AV core platform services repository (application-layer integration package).

### Generated image artifacts

After `make build`, image files are written to `output/images/`:

- `output/images/sdcard.img`: Full Raspberry Pi SD card image (flash this to microSD).
- `output/images/rootfs.ext2`: Root filesystem image.
- `output/images/rootfs.ext4`: Symlink to `rootfs.ext2`.
- `output/images/boot.vfat`: Boot partition image containing firmware, kernel, and DTBs.
- `output/images/Image`: Linux kernel image.
- `output/images/bcm2711-rpi-4-b.dtb`: Device tree for Raspberry Pi 4 Model B.
- `output/images/bcm2711-rpi-400.dtb`: Device tree for Raspberry Pi 400.
- `output/images/bcm2711-rpi-cm4.dtb`: Device tree for Compute Module 4.
- `output/images/bcm2711-rpi-cm4s.dtb`: Device tree for Compute Module 4S.
- `output/images/rpi-firmware/`: Raspberry Pi firmware files copied into the boot image.

## Prerequisites (Ubuntu/WSL)

Install host build dependencies:

```bash
sudo apt update
sudo apt install -y build-essential git wget rsync file bc cpio unzip python3
```

You may also need additional packages depending on your host and selected Buildroot packages.

## Quick start

1. Build image:

```bash
make build
```

2. Resulting image files are in:

- `output/images/`
- `output/images/sdcard.img` (if generated by selected config)

## Common workflow

```bash
# Clone/update Buildroot and apply this repo's defconfig
make defconfig

# Change Buildroot options
make menuconfig

# Build everything
make build

# Save minimized config back to configs/pi4_64_defconfig
make savedefconfig
```

## Pinning Buildroot

By default, the Makefile checks out `master`. To pin to a tag or commit:

```bash
make BUILDROOT_REF=2024.02.11 build
```

## Flash image to SD card

Use your preferred imaging tool (for example, Raspberry Pi Imager or `dd`) to write `output/images/sdcard.img` to a microSD card.

## Notes

- This setup starts from Buildroot's upstream Raspberry Pi 4 64-bit defconfig.
- Kernel, firmware, and image generation settings are maintained in `configs/pi4_64_defconfig`.

## Autonomous Stack Baseline

The current `configs/pi4_64_defconfig` is tuned as a practical autonomous-vehicle edge baseline for Raspberry Pi 4:

- Perception and media: OpenCV4, GStreamer, FFmpeg
- Runtime and scripting: Python 3, pip, NumPy, PyYAML
- Vehicle and sensor I/O: can-utils, libsocketcan, i2c-tools, libgpiod2, gpsd
- Messaging and telemetry: Mosquitto (MQTT), ZeroMQ
- Networking and diagnostics: iproute2, ethtool, tcpdump, iperf3
- Time synchronization: chrony, linuxptp

This is a foundation image for autonomy development, not a full AD stack (for example ROS 2/Autoware integration is still a next step).

## av_services integration

This repository integrates `https://github.com/rudupa/av_services` as a git submodule under `external/av_services` and also deploys it into the root filesystem through a Buildroot external package.

Component equivalence summary:

- This repo provides platform/base-image layers (BSP, kernel, OS, middleware packages).
- `av_services` provides higher-level AV platform services (gateway, health, logger, OTA, orchestrator).
- Therefore they are complementary, not equivalent.

Submodule workflow:

```bash
# Initialize submodules after clone
make submodules

# Re-sync to recorded submodule commits
make submodules-update
```

Typical end-to-end integration flow:

1. Build and flash this Buildroot image to target hardware.
2. Confirm AV service sources are present on target at `/opt/av_services`.
3. Build and run services from `/opt/av_services` on target as needed.

Buildroot packaging notes:

- The package is defined in `br2_external/package/av_services/`.
- `BR2_PACKAGE_AV_SERVICES=y` is enabled in `configs/pi4_64_defconfig`.
- During image build, the repository contents are installed into `/opt/av_services` in the target rootfs.
